### 【AIへの質問】pythonで推奨されているコーディング規約を教えてください。

### 【AIからの回答】
Pythonで推奨されているコーディング規約は、主に **PEP 8**（Python Enhancement Proposal 8）にまとめられています。PEP 8はPython公式のスタイルガイドであり、読みやすく保守しやすいコードを書くための指針となります。

以下に主要なポイントをまとめます。

---

## 🧾 基本方針（PEP 8）

### 1. **インデント**

* スペース4つを使う（タブではなくスペース）。
* ブロックの入れ子が深くなるときもスペース4つずつ。

### 2. **行の長さ**

* 1行の長さは **79文字以下** が推奨。
* コメントやdocstringは **72文字以下** を推奨。

### 3. **空行**

* トップレベル関数・クラスの間には **2行** 空ける。
* メソッドの定義内などでは **1行** 空けると見やすくなる。

### 4. **インポート**

* 3つのグループに分け、順番に並べる：

  1. 標準ライブラリ
  2. サードパーティライブラリ（例: `requests`）
  3. 自作モジュール
* 各グループの間には **空行** を1行入れる。

```python
import os
import sys

import requests

import mymodule
```

| 種類    | 命名規則            | 例                   |
| ----- | --------------- | ------------------- |

| モジュール | `snake_case.py` | `my_module.py`      |
| パッケージ | `snake_case`    | `my_package`        |

## 📘 コメントとDocstring

* **行コメント** はコードの右側に書くより、上に独立させた方がよい。
* **docstring**（`""" """`）をクラス、関数、モジュールに付ける。

```python
def add(a, b):
    """2つの数値を加算して返す"""
    return a + b
```

## 🔍 可読性のための工夫

* `== None` ではなく `is None` を使う。
* `not x in y` より `x not in y`。
* 冗長な括弧を避ける。

### 【AIからの回答】
Pythonで推奨されているファイル名（スクリプトやモジュールの名前）の命名規則は、**PEP 8**および**実務上の慣習**に基づいて以下のようにされています。

---

## 🧾 Pythonファイル名の命名規則（PEP 8準拠）

### 🔹 基本ルール

| 要素      | 命名規則            | 例                              |
| ------- | --------------- | ------------------------------ |
| モジュール名  | `snake_case.py` | `my_module.py`                 |
| パッケージ名  | `snake_case`    | `my_package`                   |
| テストファイル | `test_*.py`     | `test_utils.py`, `test_api.py` |

---

### 🔹 具体的なポイント

1. **英小文字とアンダースコア（snake\_case）を使用**

   * `camelCase.py` や `PascalCase.py` は避ける
   * 例: `data_loader.py`, `api_client.py`

2. **短く、意味が分かりやすい名前にする**

   * 例: `utils.py` よりも `string_utils.py` など用途を明確に

3. **数字から始めない**

   * `2fast.py` → ❌（無効な識別子になる可能性）

4. **特殊文字やスペースを避ける**

   * `my-script.py` → ❌（ハイフンはimportに不適）

---

### 🔹 特殊ケース（Pythonが自動的に使う名前）

| ファイル名         | 用途                             |
| ------------- | ------------------------------ |
| `__init__.py` | パッケージの初期化ファイル（Python 3.3以降省略可） |
| `__main__.py` | パッケージをスクリプトとして実行するためのエントリーポイント |
| `setup.py`    | パッケージのインストール設定用ファイル            |

## 🥇 迷ったらこれ：**FastAPI**（おすすめ）

### 特徴

* Python 3.7以降の型ヒントを活用して自動で**ドキュメント生成（Swagger/OpenAPI）**
* 非同期対応（`async/await`）によって高速
* バリデーションやスキーマ管理に**Pydantic**を使用
* テストもしやすく、拡張性が高い

### こんな人に向いてる

* モダンなPythonで、堅牢なREST APIやJSON APIを作りたい
* 自動ドキュメントが欲しい
* 型安全なコードが好き

---

## 🧰 用途ごとの選び方

| 用途例                    | おすすめ                        |
| ---------------------- | --------------------------- |
| 学習・軽量なAPI作成            | Flask, FastAPI              |
| 大規模サービス（管理画面やDBとの連携あり） | Django + DRF                |
| 高速・非同期・型付きAPIを重視       | FastAPI                     |
| リアルタイム通信               | Tornado, FastAPI（WebSocket） |

## 🔧 FastAPIの最小例

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/")
def read_root():
    return {"message": "Hello, API!"}
```

起動（`uvicorn`が必要）:

```bash
uvicorn main:app --reload
```

ブラウザで `http://127.0.0.1:8000/docs` にアクセス → 自動生成APIドキュメントが表示されます。
